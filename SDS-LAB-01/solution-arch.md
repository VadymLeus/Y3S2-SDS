## 1. Вступ
SpendWise — це веб-застосунок, створений для контролю доходів та витрат користувача. Його мета — забезпечити зручне середовище для введення, редагування, перегляду та фільтрації фінансових записів, з базовими функціями безпеки.

## 2. Огляд Архітектурних Рішеннь
Архітектура складається з двох основних частин: frontend (React) та backend (Node.js з Express), що взаємодіють через REST API. Дані зберігаються в базі MySQL. Використовується модифікований шаблон MVC.

## 3. Бізнес-Кейс
Потреба в особистому фінансовому контролі, а також відсутність простих рішень для обліку доходів/витрат із гнучким фільтруванням і легким доступом для користувачів без технічних знань.

## 4. Відповідність Принципам та Вимогам

### Функціональні вимоги:
- Реєстрація/авторизація користувача
- Додавання/редагування/видалення записів
- Пошук і фільтрація за категорією, назвою, сумою
- Управління профілем користувача

### Нефункціональні вимоги:
- Безпечне зберігання паролів (bcryptjs)
- Простий, інтуїтивно зрозумілий інтерфейс

## 5. Проєктування Рішення на Високому Рівні
- Frontend: React, React Router DOM
- Backend: Node.js, Express
- БД: MySQL
- REST API між frontend та backend

## 6. Технологічна Архітектура

### Frontend:
- React SPA
- Axios для HTTP-запитів
- React Router DOM для навігації
- Toastify для сповіщень

### Backend:
- Node.js + Express
- Поділ на маршрути, контролери, моделі

### Додаткові технології:
- CSS-модулі
- bcryptjs
- Font Awesome

## 7. Інтеграційна Архітектура
- REST API для взаємодії клієнта і сервера
- Axios як інструмент HTTP-комунікації

## 8. Архітектура Даних
База даних SpendWiseDB містить 3 основні таблиці:
1. users — користувачі
2. incomes — доходи
3. expenses — витрати

Таблиці зв’язані через user_id. При видаленні користувача можливе каскадне видалення пов’язаних записів.

## 9. Архітектура Безпеки
- Хешування паролів та кодових слів (bcryptjs)
- Авторизація через email/пароль
- Повідомлення про помилки на клієнті

## 10. Інфраструктурні Вимоги
- Сервер з Node.js
- База даних MySQL
- Хостинг для frontend (Vercel)

## 11. Нефункціональні Вимоги
- Час відгуку < 2 секунд
- Надійність: повідомлення про помилки
- Масштабованість: гнучка структура MVC

## 12. Стратегія Переходу та Впровадження
1. Розгортання backend та frontend
2. Ініціалізація бази даних
3. Проведення тестування функціоналу
4. Публікація на GitHub, деплой

## 13. Ризики та Шляхи Їх Усунення

| Ризик                        | Шлях усунення                           |
